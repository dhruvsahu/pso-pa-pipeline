# PsO Prior Authorization Access Quality Pipeline

A pipeline that reads healthcare payer Prior Authorization (PA) policy PDFs, extracts structured access criteria using LLMs, scores them against FDA prescribing label baselines, and produces a ranked access quality report across payers and brands.

Built for the **PsO ADS Track Hackathon**.

---

## What It Does

For each `(PDF, brand)` pair:
1. Parses the PDF page-by-page
2. Runs 5 specialized extractors (age, step therapy, authorization, utilization, clinical access)
3. Each extractor retrieves relevant pages via keyword search, ranks them by signal density, and sends the top context to an LLM
4. Scores the payer policy against the FDA label baseline (50 = FDA parity)
5. Saves results incrementally with checkpoint support

**Output:** A flat CSV/Excel with one row per `(payer, brand)` containing all extracted parameters and an access quality score.

## Quickstart

### Step 1 — Install & configure

```bash
# Install dependencies
pip install -r requirements.txt

# Set up your API key
cp .env.example .env
# Then open .env and replace the placeholder with your actual key
```

The pipeline auto-selects the backend — add **one** key:

| Key | Provider | Notes |
|-----|----------|-------|
| `GROQ_API_KEY` | Groq — llama-3.3-70b | Recommended for presentation |
| `GEMINI_API_KEY` | Gemini 3.1 Flash Lite (`gemini-3.1-flash-lite`) | Free tier: 15 RPM, 500 RPD |

No key at all → falls back to local **Ollama** (must have Ollama running).

---

### Step 2 — Interactive UI (single PDF)

The fastest way to try the pipeline. Drag in any PA policy PDF, pick a brand, and watch results stream in live as each extractor finishes.

```bash
python app.py
```

Then open **http://localhost:5000** in your browser.

1. **Drag & drop** a PA policy PDF onto the upload area
2. **Select a brand** from the dropdown (e.g. STELARA, TREMFYA, SKYRIZI)
3. Click **Analyse** — results stream in real time, one extractor at a time:
   - Age restriction
   - Step therapy requirements
   - Authorization duration
   - Quantity limits
   - TB test / specialist / precertification
   - Access quality score + category
4. Click the **Download Results** button to retrieve the cumulative CSV

Each analysis appends a row to `outputs/results.csv`. The file accumulates across runs so you can analyse multiple PDFs in sequence, then download the combined report. No uploaded PDF is stored — the file is processed in memory and discarded after the run.

---

### Step 3 — Bulk batch run (79 PDFs)

For processing the full dataset. Results are saved incrementally — if the run is interrupted, it resumes automatically from the last completed entry.

```bash
# Place PDFs in the input folder
Sample_PsO_ADS_Track/

# Run the full batch (skips already-completed entries)
python run_full_pipeline.py

# Format results to CSV + Excel once complete
python result_formatter.py
```

Outputs written to `outputs/`:
- `final_access_results.json` — full nested results (checkpoint file)
- `final_access_results.csv` — flat table, one row per (payer, brand)
- `final_access_results.xlsx` — same as CSV, Excel format

## Scoring Model (scorer v2.2)

Start at **50** — the neutral / unknown baseline. Each axis is **tri-state**: a restriction *present*
beyond the FDA label **deducts**; a restriction *confirmed absent* (explicit `"No"` / empty list /
confirmed 0) earns a **small +2 confirmed-open credit**; an *unknown* value (`"NA"`/missing) does
**nothing**. Two terms that are *strictly more permissive than the FDA label* earn larger credits
(age younger than FDA +5, TB waived +3). Final score is clamped to `[0, 100]`.

| Restriction (beyond FDA) | Deduction | Confirmed-absent credit |
|--------------------------|-----------|--------------------------|
| Brand step therapy | −10/step (max −30) | — *(both step counts confirmed 0 → +2)* |
| Generic step therapy | −5/step (max −15) | *(see above)* |
| Mandatory phototherapy step | −5 | +2 (explicit "No") |
| Specialist restriction | −8 | +2 (empty list) |
| Reauthorization required | −5 | +2 (explicit "No") |
| Quantity limits | −5 | +2 (empty list) |

| Strictly-better-than-FDA credit | Bonus |
|---------------------------------|-------|
| Age less restrictive than FDA | +5 |
| TB test waived where FDA expects it | +3 |
| TB test required where FDA does *not* expect it | −3 (deduction) |
| Age more restrictive than FDA | −5 (deduction) |

A missing/unextracted value (`"NA"`) is **neutral** on every axis — so a verified-open policy scores
above an unextracted one, but an unextracted policy is never inflated.

**Access categories** (anchors 0/25/50/75/100; half-open intervals):

| Score | Category |
|-------|----------|
| `[75, 100]` | Preferred Access *(not reachable for this dataset — see scale note)* |
| `[50, 75)` | FDA Parity |
| `[25, 50)` | Restricted Access |
| `[0, 25)` | Highly Restricted |

Observed range across 79 real-world PA policies (scorer v2.2, full re-extraction): **12–52, average 31.8, median 32**.
Category split: **23 Highly Restricted, 52 Restricted Access, 4 FDA Parity, 0 Preferred Access**.
Calibration is pinned by `test_scoring.py` (all-`"NA"` ≈50, confirmed-open ≈60, maximally restrictive
≤10, most-permissive ≈68, plus parsing guards for upper-bound age and reauth casing). Figures are
reproducible: `python rescore.py` re-scores every row from its stored extraction values and
`python result_formatter.py` regenerates the CSV/XLSX.

> **Note on the scale (why scores cap ~68 and never reach Preferred).** A prior-authorization policy
> only *adds* restrictions versus the FDA label, and for these PsO brands the FDA baseline already
> imposes no step therapy / quantity limit / specialist / reauthorization. So a policy can climb
> above 50 only via small confirmed-open credits (+2/axis, max +10) plus the two better-than-FDA
> terms (age +5, TB +3) → a practical ceiling of ~68. The objective's **75 ("preferred") and 100
> ("best access against all competitors") anchors are not reachable from the extracted parameters**
> for this dataset — they would require a competitive/formulary-positioning signal the pipeline does
> not extract. This is a deliberate, documented choice (**Option A**); see `docs/ADR.md` ADR-016 for
> the alternatives considered (cohort-relative normalization, generosity signals,
> competitive-positioning extraction) and why the faithful vs-FDA model was chosen.

## LLM Backends

Configured via `.env`. Zero code changes required to switch:

| Provider | Use case | Key |
|----------|----------|-----|
| `gemini` | Development (Gemini 3.1 Flash Lite — `gemini-3.1-flash-lite`, 15 RPM) | `GEMINI_API_KEY` |
| `groq` | Presentation (llama-3.3-70b, 12K TPM) | `GROQ_API_KEY` |
| `ollama` | Local / offline fallback | — |

The Gemini throttler (rolling 60s window, max 12 RPM) is built into `ModelRouter` — no manual rate limiting needed.

---

## Checkpoint & Recovery

Results are saved to `outputs/final_access_results.json` after **every drug**. If the run crashes:

```bash
# Just re-run — it picks up from the last saved entry
python run_full_pipeline.py
```

To force a drug to reprocess, delete its entry from the JSON first.

## Debug Files

Every extractor writes the context it sent to the LLM before each call:

```
debug/debug_step_therapy_STELARA_378692-5003182.txt
debug/debug_authorization_TREMFYA_66156-4274314.txt
```

Use these to diagnose extraction failures — they show exactly which pages were retrieved and in what order.
